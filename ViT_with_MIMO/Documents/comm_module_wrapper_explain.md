# Detailed Documentation — `comm_module_wrapper.py`

---

## General Overview

`comm_module_wrapper.py` defines the `CommModuleWrapper` class, an **advanced adapter** that wraps a `CommModule` object, making it compatible with a PyTorch `nn.Sequential` pipeline while keeping all the advanced features of the communication module active.

The application context is **split learning**: a Vision Transformer model (ViT/DeiT) is divided into two parts (client and server), and intermediate features are transmitted via a simulated communication channel. `CommModuleWrapper` is inserted exactly at this junction.

---

## File Structure

```
CommModuleWrapper (nn.Module)
│
├── __init__()              # Initialization: builds CommModule and internal attributes
├── set_score_source()      # Connects the upstream compressor block as score source
├── _get_selection_scores() # Extracts importance scores for resource allocation
├── forward()               # Main pipeline: extracts scores → runs CommModule → saves stats
├── get_last_info()         # Safe accessor to the last forward's statistics
└── reconfigure()           # Partial channel reconfiguration (e.g. SNR sweep)
```

## Dependencies and imports

```python
import torch
import torch.nn as nn
from .comm_module import CommModule
```

The file only imports standard PyTorch modules and the `CommModule` class from the same package. There are no additional external dependencies.

## `CommModuleWrapper` Class

### Purpose

The class acts as an **intermediary** between the client backbone (ViT encoder) and the underlying `CommModule`. Its three main tasks are:

1. **Compatibility with `nn.Sequential`**: the `forward` method accepts and returns a tensor, without exposing tuples or complex data structures outwards.
2. **Wiring of importance scores**: reads the attention scores generated by the upstream compressor block and passes them to `CommModule` to guide power, stream, or transmission mode allocation.
3. **Exposing utilities**: provides `get_last_info()` for channel statistics logging and `reconfigure()` to modify channel parameters at runtime.

---

### `__init__(self, input_dim: int, config: dict)`

```python
def __init__(self, input_dim: int, config: dict):
    super().__init__()
    self.comm = CommModule(input_dim=input_dim, config=config)
    self._score_source = None
    self.last_info: dict = {}
```

**Parameters:**
- `input_dim` (`int`): dimension of embedding D (e.g., 192 for DeiT-tiny). Corresponds to the feature axis dimension in the `[B, N, D]` tensor.
- `config` (`dict`): full configuration dictionary, passed directly to `CommModule`. Must contain the top-level `'comm'` key.

**What it does:**
- Instantiates `CommModule` with the received parameters and saves it as `self.comm`. This is the module that actually performs coding, transmission, and decoding of features.
- Initializes `self._score_source = None`: a reference to the upstream compressor block, which will form the connection via `set_score_source()`. As long as it's `None`, scores are unavailable, and `CommModule` utilizes uniform allocation.
- Initializes `self.last_info = {}`: empty dictionary that will house the statistics returned by `CommModule` (e.g., effective SNR, number of transmitted symbols, rate, etc.) after each forward pass.

---

### `set_score_source(self, compressor_module)`

```python
def set_score_source(self, compressor_module) -> None:
    self._score_source = compressor_module
```

**When it is called:** only once, during model construction in `proposal.build_model()`, after both the compressor and the wrapper have been instantiated.

**What it does:** registers the `Compress_Batches_and_Select_Tokens_Block_Wrapper` block as the importance scores source. This block must expose at least one of the following attributes:
- `last_adc_scores`: scores aligned with the ADC mechanism (highest priority).
- `block.attn.class_token_attention`: raw attention scores of the CLS token (fallback).

Through this connection, `CommModuleWrapper` can access scores at every forward pass without modifying the `forward` method's signature itself, preserving compatibility with `nn.Sequential`.

---

### `_get_selection_scores(self, x: torch.Tensor)`

This is the most elaborate method of the class. Its task is to extract, validate, and align importance scores to pass over to `CommModule`.

#### Execution flow

```
self._score_source is None?
    └─ Yes → return None (uniform allocation)
    └─ No →
        1. Try last_adc_scores (shape check: [B, N])
        2. Fallback to block.attn.class_token_attention
        3. Validity check (None, non-Tensor)
        4. Detach + device/dtype movement
        5. CLS token strip → patch_scores [B, N-1]
        6. Edge case check (N-1 == 0)
        └─ return patch_scores [B, N-1]
```

#### Detailed Score Alignment Contract

This is the most critical point of the method. The `class_token_attention` attribute has shape `[B, N]`, where:
- **position 0**: CLS token attention row — **not a patch score**.
- **positions 1…N-1**: patch token attention scores.

`CommModule._resolve_power_scores()` expects a shape score of `[B, N_patches]` = `[B, N-1]`. If passing the `[B, N]` tensor without tuning, the internal slice of `CommModule` would generate:
```
scores[:, :n_patches]  →  [CLS_score, patch_1, …, patch_{N-2}]
```
Which is **wrong** (off-by-one: the CLS goes in as a patch score and the last patch is missed).

The fix applied is:
```python
patch_scores = scores[:, 1:]   # [B, N-1]
```
This drops the CLS score and yields just the correctly aligned set of patch token scores.

#### Score Source Priorities

| Priority | Source | Condition |
|---|---|---|
| 1 (high) | `self._score_source.last_adc_scores` | 2D Tensor, `shape[0] == B`, `shape[1] == N` |
| 2 (fallback) | `self._score_source.block.attn.class_token_attention` | Attribute present and not None |
| 3 (final fallback) | `None` | Any error or absent attribute → uniform allocation |

### `forward(self, x: torch.Tensor) -> torch.Tensor`

```python
def forward(self, x: torch.Tensor) -> torch.Tensor:
    selection_scores = self._get_selection_scores(x)
    out, info = self.comm(x, selection_scores=selection_scores)
    self.last_info = info
    return out
```

**Input:** tensor `x` of shape `[B, N, D]` coming from the client backbone (partial ViT encoder), where:
- `B` = batch size
- `N` = token count (1 CLS + N-1 patches)
- `D` = embedding dimension

**Output:** tensor `out` of shape `[B, N, D]`, identical to the input shape. Houses the reconstructed features following the simulated channel transmission.

**Internal flow:**
1. Calls `_get_selection_scores(x)` obtaining importance scores `[B, N-1]` or `None`.
2. Calls `self.comm(x, selection_scores=selection_scores)`: `CommModule` conducts the coding, the simulated channel transfer (adding noise, compression, score-driven resource allocation) and the decoding. Returns a tuple `(output_tensor, statistics_dictionary)`.
3. Stores `info` within `self.last_info` to make it accessible to the training loop via `get_last_info()`.
4. Emits only the `out` tensor, ensuring `nn.Sequential` compatibility.

---

### `get_last_info(self) -> dict`

```python
def get_last_info(self) -> dict:
    return dict(self.last_info)
```

**What it does:** yields a **copy** of the `last_info` dict updated from the newest forward pass. Restoring a copy (rather than a reference) acts as a defensive maneuver: prevents the client from accidentally changing internal states.

**Routine usage within training loop** (found in `main.py`):
```python
if hasattr(model, "channel") and hasattr(model.channel, "get_last_info"):
    batch_stats = model.channel.get_last_info()
    for k, v in batch_stats.items():
        epoch_stats[k] = epoch_stats.get(k, 0.0) + v
```
Statistics logged from every batch are later averaged out for the entire epoch.

**Primary statistics residing in `last_info`** (created by `CommModule`):
- `snr_db`: Effectual SNR deployed for transmission
- `tokens_sent`: Sum of sent tokens
- `mimo_L`: Quantity of MIMO symbols
- `rate_symbols`: Symbols rate
- `mode_alloc_top_imp_frac` / `stream_alloc_top_imp_frac`: Share of resources allocated precisely to the top tokens

---

### `reconfigure(self, config_snippet: dict) -> None`

```python
def reconfigure(self, config_snippet: dict) -> None:
    self.comm.reconfigure(config_snippet)
```

**What it does:** routes reconfiguration calls directly toward `CommModule.reconfigure()`, supporting runtime channel adjustments **without reinitializing the full model**.

**Primary use case — SNR sweep at validation span** (located in `main.py`):
```python
for snr in snr_sweep:
    model.channel.reconfigure({'channel': {'snr_db': snr}})
    v_loss, v_acc, v_stats = validation_phase(model, val_data_loader, ...)
```
For each single SNR step from the sweep list, the channel converts effectively on-the-fly, validates data appropriately, and advances. This engineers performance-vs-SNR graph generation dodging reinitialization latency overheads.

**Architecture of `config_snippet`:** serves as a sub-dictionary to the base `'comm'` configuration. Selectively overwrites only active keys while everything else persists unaltered.

## Full Data Pipeline

```
Client Backbone (Partial ViT)
        │
        │  x: [B, N, D]
        ▼
CommModuleWrapper.forward(x)
        │
        ├──► _get_selection_scores(x)
        │         │
        │         ├─ last_adc_scores [B, N]  →  strip CLS  →  [B, N-1]
        │         └─ class_token_attention [B, N]  →  strip CLS  →  [B, N-1]
        │                    │
        │              selection_scores [B, N-1]  (or None)
        │
        ▼
   CommModule(x, selection_scores)
        │
        │  Encoding → Channel (noise + allocation) → Decoding
        │
        ├──► out: [B, N, D]   (reconstructed features)
        └──► info: dict        (channel statistics)
        │
        ▼
   self.last_info = info
        │
        ▼
   return out: [B, N, D]
        │
        ▼
Server Backbone (Second half ViT)
```

## Relationship with `main.py`

`CommModuleWrapper` is the module that `main.py` accesses via `model.channel`. The three primary interactions are:

| Code within `main.py` | Required method | Motivation |
|---|---|---|
| `hasattr(model.channel, "get_last_info")` + call | `get_last_info()` | Logging data per batch during iteration rounds |
| `model.channel.reconfigure({'channel': {'snr_db': snr}})` | `reconfigure()` | Iterative SNR updates testing |
| `hasattr(model, "channel")` | — | Generic safeguard preventing accidental interactions devoid of core routines |

## Design Observations

- **Maintains `nn.Sequential` API stability**: The `forward` handler accepts operations solely in tensor constructs, seamlessly aligning exactly as alternative sequential framework tiers coordinate. All complexity (scores extraction, parameter tweaking telemetry) gets sequestered underneath cleanly.
- **Fail-safes established uniformly**: Areas suspect to bad parameters—for example, improperly matched tokens sets processing alongside misidentified units—all carry default `None` triggers that automatically deploy basic setups, bypassing unannounced crashing errors entirely.
- **Role delegation bounds**: The wrapper retains blindness regarding the complexities operating behind `CommModule` routines (signal modulation noise manipulation structures). Operating cleanly to pass parameters faithfully upstream directly back into generalized configurations effectively.

# Documentation — `analyze_mimo_scenarios.py`

## General Purpose

This Python script is an **adaptive experiment results aggregator** produced by multiple training runs in a split learning system with a MIMO channel. It reads the metrics files generated by each run, extracts the configuration from Hydra metadata, automatically classifies parameters into constants and variables, and produces a single structured JSON file containing all results with constant parameters separated into a metadata section.

The overall logical flow is as follows:

```
Results directory (multirun or custom)
        │
        ▼
Recursive search for best_training_results.json / final_training_results.json
        │
        ▼
For each found file:
  ├── Read metrics from JSON
  ├── If re-analyzing: Select best epoch from history (new logic)
  ├── Extract config from .hydra/overrides.yaml
  └── Fallback: ascend the tree looking for "key=value" folders
        │
        ▼
Adaptive parameter analysis:
  ├── Constants (1 unique value)  → output JSON metadata 
  └── Variables (>1 unique values) → "variable_parameters" field
        │
        ▼
Write flat output with automatic versioning:
  analysis_results/analysis_flat/comparison_results.json
```

## Functions

### `parse_value(value_str: str) -> Any`

**Purpose:** Converts a string to a Python numeric type, if possible.

**How it works:** First attempts conversion to `float` (if the string contains a decimal point), then to `int`. If both conversions fail with `ValueError`, it returns the original unchanged string. This function is extracted as a standalone helper compared to the previous version, where the same logic was inline and duplicated in `extract_config_from_overrides`.

```python
parse_value("10.0")  # → 10.0 (float)
parse_value("4")     # → 4 (int)
parse_value("mmse")  # → "mmse" (str)
```

---

### `extract_config_from_overrides(folder_path: Path) -> Dict[str, Any]`

**Purpose:** Extracts the configuration parameters of a single run by reading the `.hydra/overrides.yaml` file that Hydra automatically generates in the output folder of each experiment.

**Detailed operation:**

1. Builds the path to `.hydra/overrides.yaml` via the `pathlib` `/` operator, more readable and cross-platform than `os.path.join`.
2. If the file exists, opens it with explicit UTF-8 encoding and deserializes it with `yaml.safe_load`.
3. Verifies that the content is not empty or `None` (possible if the file is present but empty) and returns `{}` in that case.
4. For each string in the `"key=value"` format, splits at the first `=` and simplifies the key to only the last segment of the pointed path (e.g. `comm.channel.snr_db` → `snr_db`).
5. The value is converted to the appropriate type via `parse_value`.
6. Error handling is now **separated by exception type**: `yaml.YAMLError` for YAML parsing errors (logged as `WARNING`) and generic `Exception` for other unexpected errors (logged as `ERROR`), ensuring more informative log messages compared to the generic `except` of the previous version.

---

### `extract_fallback_config(folder_path: Path) -> Dict[str, str]`

**Purpose:** Recovers configuration parameters from the folder name itself, in case `.hydra/overrides.yaml` is unavailable. Works as a fallback mechanism for runs not managed by Hydra or with custom output structures.

**How it works:** Checks if the folder name contains the `=` character. If so, it splits at the first `=` and returns a dictionary with the extracted key-value pair (e.g. a folder named `snr_db=10.0` produces `{"snr_db": "10.0"}`). Values remain strings (not converted).

This function is a **generalization** over the previous version, which only searched for the hardcoded `"comm="` pattern. Now any parameter in the `key=value` format in the folder name is properly extracted.

---

### `select_best_epoch_from_history(history: Dict[str, Any], criterion: str, snr_index: int = 0) -> Dict[str, Any]`

**Purpose:** Identifies the "best" epoch from the full training history based on a specific performance criterion and samples all associated metrics for that epoch.

**Supported Criteria:**
- **`average`**: Selects the epoch with the highest average validation accuracy across all SNR levels.
- **`max_noise`**: Selects the epoch with the highest validation accuracy at the lowest SNR level (most challenging condition).
- **`min_noise`**: Selects the epoch with the highest validation accuracy at the highest SNR level.
- **`snr_index`**: Selects the epoch based on the accuracy at a specific SNR index (e.g., the 2nd SNR level in the sweep).
- **`last`**: Simply takes the final recorded epoch.

**Operation:**
1. Parses the validation accuracy lists corresponding to the chosen criterion.
2. Finds the index of the maximum value (or the last index if `last` is chosen).
3. Constructs a `best_stats` dictionary by extracting the values of all other metrics (loss, cost, etc.) at that specific index.
4. Maps the original key names (e.g., `Val accuracies`) to "best" names (e.g., `best_val_accuracy`) for consistency with the standard output format.

---

### `parse_results_to_dict(base_dir: Path, force_reanalyze: bool = False, criterion: str = "average", snr_index: int = 0) -> List[Dict[str, Any]]`

**Purpose:** Recursively scans the base directory, collects all training result files, and combines the metrics with the run's configuration for each.

**Detailed operation:**

1. **File search:** Uses `Path.rglob()` to search for `best_training_results.json` by default. If `--force_reanalyze` is active, it specifically searches for `final_training_results.json` to access the full history.

2. **For each found file:**
   - Loads the JSON metrics.
   - **Re-analysis logic:** If `force_reanalyze` is true or if the file contains full history lists, it calls `select_best_epoch_from_history` to determine the best epoch and statistics according to the specified `--selection_criterion`.
   - Calls `extract_config_from_overrides` on the file's folder.

3. **Generalized fallback:** If the configuration extracted by Hydra is empty (`{}` dictionary), the script ascends the folder tree starting from the file's folder towards the `base_dir`. At each level, it calls `extract_fallback_config` on the current folder name and adds the found key-value pairs to the configuration, **without overwriting** existing keys (thanks to the `if k not in config` check). The ascent stops when reaching the `base_dir` or encountering a folder with a special name (`"results"`, `"multirun"`, `"analysis_results"`). This mechanism is more robust and generic than the previous fallback, which only looked for `"comm="` in the name and didn't ascend the tree.

4. Builds each entry as a dictionary with `"source_path"` (string, not `Path` object, to ensure JSON serializability), `"conditions"`, and `"best_epoch_stats"`.

---

### `analyze_parameters(all_data: List[Dict[str, Any]]) -> Tuple[List[str], List[str]]`

**Purpose:** Analyzes the set of all collected runs, classifies each parameter as constant or variable, and returns the two alphabetically sorted lists.

**How it works:** Collects all parameter names present in any entry, then for each, builds the set of unique values (converted to strings for type consistency). A parameter is classified as:

- **Constant** (`hierarchy_params`): has at most 1 unique value across all runs. The threshold is `<= 1` (not `== 1` as in the previous version), which properly handles parameters present in some runs but not others — if a parameter appears in only one run, it is considered constant.
- **Variable** (`comparison_params`): has more than 1 unique value, i.e., it changes across runs.

Both lists are sorted with `sorted()` to ensure deterministic output independent of dictionary iteration order.

Compared to the previous version, this logic has been extracted into a dedicated function with a typed signature, instead of being inline in `main()`.

---

### `save_flat_results(args, all_data, hierarchy_params, comparison_params)`

**Purpose:** Saves the aggregated results in a **flat** (single-file) output structure instead of the folder tree structure of the previous version, incorporating constant parameters as metadata in the output JSON file.

**Detailed operation:**

1. **Automatic versioning:** Builds the output path as `{base_output_dir}/analysis_flat/` by default. If `--force_reanalyze` is used, the suffix changes to `analysis_reanalyze_{criterion}` (e.g., `analysis_reanalyze_max_noise`). If the folder already exists, it appends an incremental numeric suffix (e.g., `_1`, `_2`).

2. **Building constant metadata:** Reads the `hierarchy_params` values from the first list entry (being constant, the value is the same for all runs). These are collected in the `constant_metadata` dictionary.

34. **Output payload structure:** The JSON written to disk has a structure that includes a `metadata` header with info about the re-analysis (if any):
   ```json
   {
       "metadata": {
           "constant_parameters": { ... },
           "variable_parameters": ["n_tx", "snr_db"],
           "total_runs": 6,
           "reanalyzed": true,
           "criterion": "max_noise"
       },
       "runs": [ ... ]
   }
   ```
   The `"metadata"` section is the most significant change from the previous version: constant parameters no longer create a folder hierarchy but are explicated as shared context in the file itself. This avoids overly long paths and simplifies programmatic reading of the file by other analysis scripts.

4. Opens the output file with explicit UTF-8 encoding and writes the JSON with 4-space indentation. Any writing errors are logged as `ERROR` without crashing the process.

---

### `main()`

**Purpose:** Entry point. Coordinates the entire sequential pipeline: argument parsing → data collection → parameter analysis → saving.

**Command-line arguments:**

| Argument | Default | Description |
|---|---|---|
| `--results_dir` | `"results"` | Path to the directory containing runs. |
| `--base_output_dir` | `"analysis_results"` | Base directory where output is written. |
| `--force_reanalyze`| `False` | Flag to ignore existing `best` files and re-calculate best epoch from history. |
| `--selection_criterion`| `"average"`| Criterion for choosing the best epoch (`average`, `last`, `max_noise`, `min_noise`, `snr_index`). |
| `--selection_snr_index`| `0` | Specified SNR index to use when `criterion='snr_index'`. |

Converts `args.results_dir` to a `Path` object before passing it downstream, ensuring compatibility with the `pathlib` API used contextually.

## Output Structure

Assuming runs with these parameters:

| Run | `model` | `dataset` | `snr_db` | `n_tx` |
|---|---|---|---|---|
| 1 | `vit_small` | `cifar10` | `5.0` | `2` |
| 2 | `vit_small` | `cifar10` | `10.0` | `2` |
| 3 | `vit_small` | `cifar10` | `20.0` | `4` |

The script would determine:
- `hierarchy_params` = `["dataset", "model"]` (constants → placed in `metadata`)
- `comparison_params` = `["n_tx", "snr_db"]` (variables → documented in `variable_parameters`)

Output is a single file:
```
analysis_results/
└── analysis_flat/
    └── comparison_results.json
```

With content:
```json
{
    "metadata": {
        "constant_parameters": {
            "dataset": "cifar10",
            "model": "vit_small"
        },
        "variable_parameters": ["n_tx", "snr_db"],
        "total_runs": 3
    },
    "runs": [
        {
            "source_path": "...",
            "conditions": { "model": "vit_small", "dataset": "cifar10", "snr_db": 5.0, "n_tx": 2 },
            "best_epoch_stats": { "val_acc": 0.82, ... }
        },
        ...
    ]
}
```
